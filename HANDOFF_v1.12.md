## What shipped, task by task

**H1 — Boot settings broadcast** (`2a477f2`). The glasses used to boot on hardcoded defaults every single session and only learn the phone's real settings if the user changed something afterward — a silent split-brain (especially bad with a manual location set). `settings-hydrated` now also emits `broadcast-settings`. Caught and fixed two other pre-existing tests that encoded the bug as intended behavior.

**H3 — Absolute geocode URLs** (`97f2483`, `9c66925`). `/api/geocode`'s two phone-side call sites used relative URLs, which fail in installed EHPK builds (WebKit rejects a relative fetch against the WebView's non-http(s) base) — the same bug class already fixed once for `/api/poi` back in v1.0.2. Now both use the exported `API_BASE`. The first test-quality pass didn't actually distinguish fixed from unfixed code (Vitest's `DEV=true` default made `API_BASE` resolve identically either way) — second commit fixes the tests to stub `DEV=false` via `vi.resetModules()` + dynamic re-import, verified genuinely red-then-green.

**M5 — Coordinate log rounding** (`e422163`). `/api/poi`'s diagnostic log was writing full-precision user lat/lng to Vercel logs. Now rounds to 2 decimals (~1km) via a new `roundCoord` helper.

**L5 — Shared server geo helpers** (`b9e6051`). `haversine`/`bearing` were duplicated between `api/poi.ts` and the client-side `src/glasses/geo.ts`. Extracted the server copy into `api/_lib/geo.ts` (kept separate from the client copy — `api/` can't import from `src/` without breaking the frontend bundle boundary).

**L6 — Drop redundant `/api/wiki` summary call** (`68d6894`). Removed the separate Wikipedia REST summary fetch; `title`/`summary` now derive from the full-extract call alone (which was already being fetched), halving upstream latency per request. The existing `404` guard already only depended on the extract call, so no new error handling was needed.

**L7 — Stale API.md URL** (`3d1b12d`). Fixed a leftover reference to the old personal-account Vercel subdomain; production has been `wander-six-phi.vercel.app` for a while.

**L2 — Shared DEV mock-coords helper** (`c2e2d04`). `VITE_MOCK_LAT`/`VITE_MOCK_LNG` parsing was duplicated verbatim in `effects.ts` and `App.tsx`. Extracted to `src/glasses/devMock.ts`, used by both.

**L3 — Skip reverse-geocode for manual fixes** (`93ddc48`). `location-acquired` used to fire a reverse-geocode call even when the user had just typed a specific address into the manual-location search — the result would eventually overwrite their chosen label with a generic reverse-geocoded string. Now skipped when `source === 'manual'`.

**M2 — Phone geolocation wall-clock race** (`d360ac4`). Mirrors the glasses side's existing 15s `Promise.race` pattern — `getCurrentPosition`'s own timeout is known-unreliable on real G2 WebViews (the callback sometimes never fires at all), and the phone side had no equivalent guard, meaning the manual-location rescue form was unreachable if it hung.

**M1 — Manual-location NAV_ACTIVE teleport guard** (`d97f898`). The 5-minute background POI refresh runs regardless of screen; if a manual location was set while the user was actively navigating (real GPS), the refresh's manual-location short-circuit could inject the pin's stale coordinates into `NAV_ACTIVE`, potentially triggering false arrival and stopping the GPS watch. `position-updated` now carries a `source: 'gps' | 'manual'` tag; the reducer ignores manual-sourced updates specifically while `NAV_ACTIVE`.

**M4 — Single cursor on POI_LIST** (`a696fed`). The firmware's native list-selection border and the app-drawn `>` cursor prefix are two independent state machines that could visibly point at different rows. **Conservative fix chosen deliberately**: disabled the native border (`isItemSelectBorderEn: 0`), kept the existing, already-tested app cursor. The alternative (trusting native selection, dropping the app cursor) was not attempted — it would need an on-device session to validate and this session had no hardware access.

**M6 — Settings fetch-storm mitigation** (`ab019ad`, `6dc3be8`). Two sub-fixes: (1) glasses-side skip the `fetch-pois` refetch when only a display-only field (`units`) changed; (2) phone-side debounce the network-triggering path of `request-location` (not the cheap manual-location/DEV-mock short-circuits). **The first pass shipped dead code** — the review caught that `bridge.ts` always forwards the complete settings snapshot, so checking which keys were *present* in the payload could never distinguish a real units-only change. Fixed by diffing *values* against current state instead, with content-aware equality for `categories`/`manualLocation`; verified by reverting to the old logic and confirming the regression test correctly goes red. **Known accepted gap, not fixed**: manual-location users get zero debounce coverage (their `request-location` short-circuits above the debounce, so every settings tick still fires an undebounced `fetch-nearby-pois`) — an explicit scope boundary from the plan, not an oversight. Also: the debounce delays even the very first boot-time `request-location` by 500ms — cosmetic, not a correctness issue.

**M3 — Minimap push serialization** (`7a4785c`). Every NAV_ACTIVE position update triggered a new unawaited `pushMinimap()` (canvas draw + PNG encode + tile fetches + BLE image push) with no guard against two overlapping — risking an out-of-order stale push, and violating the platform's "no concurrent image sends" rule. Added `createInFlightGuard()`, a small standalone tested unit (drops a new push while one is in flight, rather than queueing), wired into the one `NAV_ACTIVE` call site. **Note for future sessions**: the original task brief assumed `bridge.test.ts` had an existing mocking seam for this — it didn't. Don't build a full mocked-SDK/bridge/fetch integration harness for something like this; extracting the core logic into a standalone tested unit and leaving the thin wiring call-site unit-test-free (consistent with `initGlasses` having zero test coverage overall, an existing condition this task didn't change) was the right-sized answer.

**H2 — Retry no longer dead-ends for route/wiki failures** (`44cf616`, `b449b3b`). This was the largest architectural fix: retrying after a failed route or wiki fetch used to transition to a bare LOADING screen with **zero effects attached** — nothing ever actually retried, and the reducer had no path back from LOADING to resume WIKI_READ/NAV_ACTIVE/POI_ACTIONS anyway. Added a `RetryContext` type capturing the originating screen + exact request params at failure time, two new `ALLOWED_TRANSITIONS` edges (`ERROR_NETWORK→POI_ACTIONS`, `ERROR_NETWORK→NAV_ACTIVE`), and rewrote `onRetry` to restore the exact originating screen and re-fire the exact failed effect — no LOADING detour at all for these two cases. **Review caught a real follow-up gap**: the original fix captured retry context from whatever screen was active when the failure *landed*, not when the fetch was *fired* — if a user backed out of `POI_ACTIONS(A)` and landed on `POI_ACTIONS(B)` before the stale `fetch-route(A)` failed, retry would show the correct-looking screen B but silently re-fire the route fetch for stale POI A. Fixed by gating `retryContext` on POI-identity match (`poi.id`/`destination.id`/`wikiTitle`); no match falls back to POI_LIST (same safe default as "user navigated away entirely"). TDD-confirmed red-then-green. **Known accepted note**: `goLoading`'s `null` parameter/branch is now dead code (no caller passes `null` post-refactor) — harmless, left as a documented cleanup opportunity, not fixed in this task.

**L9 — Remove unused `_bridge` param** (`47e5b04`). `translateGlassesEvent`'s 4th parameter was dead — never read in the function body. Pure mechanical signature shrink across 1 production + 35 test call sites (drifted up from the ~30 estimated when the plan was written).

**L4 — Post-build bundle whitelist check** (`e9bb0f8`, `c8f9fdf`). The existing `network-whitelist.test.ts` only scans hand-authored source, not the built `dist/` bundle — a URL baked into a third-party dependency would slip through undetected until actual store submission (the same class of gap that got v1.10 rejected). New `scripts/check-bundle-whitelist.mjs`, wired into `pack` between `build` and `evenhub pack`. **First pass failed on every build** — react-dom bakes in 4 W3C XML/SVG/MathML namespace URI strings (used with `createElementNS`, never fetched) that the naive URL-regex scanner flagged as violations every single time, which would have trained anyone running `pack` to expect-and-ignore a failure. Fixed with a narrow, exact-match exclusion set (not added to `app.json`'s real whitelist, since they aren't network endpoints) — independently re-verified to exit 0 on a clean build while still catching a genuinely new unwhitelisted URL.

**L1 — Bridge-backed settings storage with safe fallback** (`0cdc2ee`). `WANDER_BUILD_SPEC.md` §17 mandates `bridge.setLocalStorage()` for persistence — "browser localStorage unreliable in Flutter WebView" — but the phone side's settings persistence was still on raw `window.localStorage`, contradicting that spec (the glasses side already complied, via favorites). `createBridgeKVStore` already existed and was already tested, just never wired in. Now `kv` upgrades to a bridge-backed store when running inside the real EvenHub host (gated on `flutter_inappwebview`, matching `main.tsx`'s exact gate), falling back to the existing browser store on any failure — additive, match-or-improve, never worse than today. The trickiest part: the boot sequence now correctly `await`s store resolution before the *first* settings/cache read, so hydration never races an in-flight bridge handshake (traced precisely during review — this is exactly the class of split-brain bug H1 fixed elsewhere in this session, and it would have been easy to reintroduce here by accident).

**L8 — Language picker** (`ff2d134`). New feature: a Language row in phone Settings, offering exactly the 20 languages `api/route.ts`'s `ORS_SUPPORTED_LANGS` supports (not the full open-ended set the server's `resolveLang` would technically accept) — every choice works end-to-end for navigation, Wikipedia, and POI fetches. Threaded through phone storage (`wander_lang`) → `lang-changed` event → `broadcast-settings` CustomEvent → glasses `handleSettingsChanged` → the glasses' `Settings.lang` field, which already existed and was already consumed everywhere but was permanently `null` because nothing on the phone side had ever set it. Two sound mid-task additions beyond the brief's literal text: extracted `parseSettingsChangedEvent` as a standalone tested unit (no prior test seam existed for the inline closure), and added `lang` to M6's `onlyUnitsChanged` value-diff conjunction (it was genuinely missing — without it, a simultaneous units+lang change would have been misclassified as "only units changed" and silently skipped a needed refetch). **`app.json`'s `supported_languages` was initially left at `["en"]`, then expanded to all 20 codes on 2026-07-04 per Steven's request** — see Open Flags below.

**Task 20 — Real user bug report: "Open in Maps"/"Website" links fail on Android** (`db14e1e`). A user reported "Page Web non disponible" / `net::ERR_UNKNOWN_URL_SCHEME` tapping "Open in Maps." Root cause: Google's Maps page redirects Android browsers to an `intent://` deep-link URL (nothing to do with cookies, despite the user's guess), which the EvenHub host's embedded WebView can't parse. This is unrelated to Wander's own glasses navigation (which never touches Google Maps at all — it's Wander's own ORS-based routing rendered on the G2 display). Fixed by reusing an already-field-tested pattern that solved the identical problem once before, on the glasses side: `defaultOpenUrl`'s `window.open(url, '_system', ...)`-first-then-`_blank`-fallback, now exported and wired into both of `NearbyTab.tsx`'s external links via `onClick`.
