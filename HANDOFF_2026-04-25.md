### Phase B — Hard wallclock timeout (`9c39f3b`)
Field test surfaced "Fetching nearby places…" stuck for 1+ minute
despite an existing 12s `AbortController` timeout. The EvenHub
WebView's `fetch` doesn't reliably honor `AbortSignal`.

Two-layer protection now:
1. AbortController on `fetch` (10s) — best-effort cancel, matches
   Vercel Hobby's 10s function budget
2. Outer `Promise.race` wallclock (12s) — ALWAYS produces a result on
   the caller side. Wraps both `fetch()` and `r.json()` since body
   streaming can hang independently

After this change, worst-case glasses experience for a hung API is
~12s → ERROR_NETWORK → user can retry.

**File:** `src/glasses/api.ts` `getJson`

### Phase C-1 — Default to all 8 categories (`9bb7f60`)
Prior default was only 5 of 8 categories (`landmark, park, museum,
religion, food` — missing `art, library, nightlife`). Until the Phone
Settings UI ships, on-glass default should be as wide as possible.

Changed:
- `src/glasses/screens/types.ts` `DEFAULT_SETTINGS.categories`
- `api/poi.ts` `parseCategories()` server-side fallback (matches
  client default so behavior is identical whether `?categories=` is
  sent or omitted)

### 2.2 [HIGH] Double-tap still unresponsive on POI_DETAIL / NAV_ACTIVE
Field test 2026-04-25 confirmed double-tap *does* work on LOADING
(opens CONFIRM_EXIT), but does NOT work on POI_DETAIL or NAV_ACTIVE.
This is inconsistent and suggests the event source differs by screen.

**Hypothesis:** LOADING screen uses `singleText` (`isEventCapture: 1`)
which delivers via `sysEvent` or `textEvent`. POI_DETAIL has a
2-container layout; the double-tap event might land on the header
container which has `isEventCapture: 0`, preventing it from reaching
the bridge.

**Where to look:**
- `src/glasses/render.ts` `renderPoiDetail` — `isEventCapture` flags
- `src/glasses/bridge.ts` `translateGlassesEvent` — Phase 0 diagnostic
  logs already in place; once log capture is solved (§2.4), check
  what `source=` is reported when double-tapping POI_DETAIL.
