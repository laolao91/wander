## Task 1: Fix B1 — `onBack` from POI_ACTIONS goes to POI_LIST

**Files:**
- Modify: `src/glasses/state.ts` (function `onBack`, case `'POI_ACTIONS'`)
- Test: `src/glasses/__tests__/state.test.ts`

### What to understand first

When a user double-taps in `POI_DETAIL`:
- Tap 1 → single-tap handler fires → screen transitions to `POI_ACTIONS`
- Tap 2 (within window) → `isManualExitTap()` returns `true` → dispatches `back`
- `back` from `POI_ACTIONS` currently returns `next(state, { name: 'POI_DETAIL', ... })`
- Result: user ends up back on `POI_DETAIL` — nothing visually useful happened

The `close` action (`'← Back'`) already handles the intentional "dismiss options menu → return to detail" case. So changing `onBack` from `POI_ACTIONS` to go to `POI_LIST` is correct: the `back` *event* (gesture) goes all the way back, the `close` *action* (deliberate menu selection) goes to detail.

`POI_ACTIONS → POI_LIST` is already in `ALLOWED_TRANSITIONS` (added in v1.2.0 for the `back` action). No transition-map change needed.

## Task 2: Fix B2 — Widen tap window + reset counter on screen change

**Files:**
- Modify: `src/glasses/bridge.ts` (constant `MANUAL_EXIT_TAP_WINDOW_MS`; function `dispatch`)
- Test: `src/glasses/__tests__/bridge.test.ts`

### What to understand first

`MANUAL_EXIT_TAP_WINDOW_MS = 350` means two taps must arrive within 350ms to be treated as a double-tap. On Android BLE, each event can take ~200ms to arrive. A user double-tapping in 150ms may see the two events arrive 350ms apart at the software level — right at or past the window boundary, causing them to register as two separate single taps.

Increasing to 500ms accommodates the latency. The risk: a tap on POI_LIST at t=0 navigates to POI_DETAIL, and a tap on POI_DETAIL at t=400ms would fall within the 500ms window and be counted as tap 2 of a double-tap, firing `back` unexpectedly. To prevent this, **reset `_clickCount` to 0 whenever the screen name changes** — cross-screen taps can never form a double-tap sequence.

## Task 3: Fix B3 — Settings refresh bypasses GPS when location is already known

**Files:**
- Modify: `src/phone/state.ts` (function `withSettingsChange`)
- Test: `src/phone/__tests__/state.test.ts`, `src/phone/__tests__/nearby.test.ts`

### What to understand first

`withSettingsChange` currently always emits `request-location`, which calls `navigator.geolocation.getCurrentPosition`. On Android WebViews, this can fail silently (PERMISSION_DENIED even with permission granted) because the EvenHub WebView context doesn't always forward the host app's location permission. The glasses update correctly because they use the EvenHub SDK's native location — completely separate path.

The fix: if `state.nearby.location` already has coordinates (which it will after the Nearby tab's first load), skip GPS re-acquisition entirely and directly emit `fetch-nearby-pois` with the stored coordinates + new settings. This is faster AND more reliable — it never depends on the Android geolocation API for a refresh.

`NearbyFetchStatus` type: `'locating'` means we're waiting for GPS; `'fetching'` means we have a location and are calling the API. When location is already known, we skip to `'fetching'`.

`fetch-nearby-pois` effect shape: `{ type: 'fetch-nearby-pois'; lat: number; lng: number; settings: Settings }`.
Note: `settings` here is `Settings` from `src/phone/types.ts` (phone-side type with `enabledCategories: readonly CategoryId[]`), **not** the glasses-side `Settings`. The `App.tsx` `runEffect` handler calls `categoryIdsToCategories(effect.settings.enabledCategories)` to convert before calling `fetchPois`.

Find `function withSettingsChange` in `src/phone/state.ts`. Replace the body with:

```typescript
function withSettingsChange(
  state: PhoneState,
  nextSettings: Settings,
): ReduceResult {
  // If we already have the user's location (acquired on the Nearby tab's
  // first load), skip GPS re-acquisition and go straight to a POI fetch
  // with the new settings. This is faster AND more reliable on Android
  // WebViews where navigator.geolocation can silently fail with
  // PERMISSION_DENIED even when the user has granted location access.
  const location = state.nearby.location
  const nearbyEffect: PhoneEffect = location
    ? {
        type: 'fetch-nearby-pois',
        lat: location.lat,
        lng: location.lng,
        settings: nextSettings,
      }
    : { type: 'request-location' }
  const nearbyStatus: NearbyFetchStatus = location ? 'fetching' : 'locating'

  return {
    state: {
      ...state,
      settings: nextSettings,
      syncStatus: 'syncing',
      syncError: null,
      nearby: { ...state.nearby, fetchStatus: nearbyStatus, errorMessage: null },
    },
    effects: [
      { type: 'persist-settings', settings: nextSettings },
      { type: 'broadcast-settings', settings: nextSettings },
      nearbyEffect,
    ],
  }
}
```
