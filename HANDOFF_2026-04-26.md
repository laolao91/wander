### Commit `86cb378` — Phase E + F + visual
**Phase E — windowed POI_LIST**
- New `displayOffset` field on `PoiListScreen`
- "More results" advances the visible window through cached items
  *instantly* (no server round-trip when there's already cache to
  scroll). Only hits the server when local cache is exhausted.
- When fetching, surfaces a LOADING screen with the new
  `LOADING_MSG_FETCH_MORE = "Loading more places..."` message
- `onPoisLoaded(mode='append')` snaps `displayOffset` to first new
  item, cursor at top of new window

**Phase F — manual double-tap detector**
- Two `CLICK_EVENT`s within 350ms promote to `request-exit` regardless
  of what the SDK reports as `eventType`. Tunable via
  `MANUAL_DOUBLE_CLICK_WINDOW_MS` in `bridge.ts`.
- Trade-off: the first click's effect still fires (we dispatch
  immediately), so a fast double-tap may briefly flash through the
  next screen before landing on CONFIRM_EXIT. UX is acceptable as a
  fallback; the alternative (deferring all single taps by 350ms) makes
  every tap feel laggy.

**Open on Phone**
- `defaultOpenUrl` now tries `window.open(url, '_system', ...)` first.
  Cordova/Capacitor and many WebView wrappers honor this target to
  route through the OS browser instead of the in-app overlay. Falls
  back to `_blank` if `_system` returns null.
- **Unverified on real hardware.** If the in-app browser still opens
  and locks the glasses, EvenHub doesn't honor `_system`; see §3.1.

**Visual: RULE shrink 40 → 28 glyphs**
- POI_DETAIL screenshot from 2026-04-26 showed `RULE` (40 × `━`)
  wrapping into a stacked double-bar on real hardware
- The G2 font isn't truly monospace; `━` is wider than alphanumerics
- 28 glyphs renders as a single line at the same visual weight
- Affects POI_DETAIL, ERROR_LOCATION, ERROR_NETWORK, ERROR_EMPTY
- LOADING screen unaffected (uses `LOADING_RULE`, separate constant)

### 3.3 [MEDIUM] Title screen layout — defer to dedicated polish pass
**Status:** Acknowledged, not fixed this session.

**Symptoms (from 2026-04-26 screenshot):**
- "WANDER" text is left-of-center, not centered
- Excessive blank space before "Fetching nearby places..."

**Why it happens:** `centeredBlock` pads with spaces at the start, but
the G2's non-monospace font means our `Math.floor((width - line.length)
/ 2)` calculation overestimates how many spaces we need to push WANDER
to center. Our `CHARS_PER_LINE = 65` constant assumes monospace.

**Fix direction (next session):**
- Either calibrate `CHARS_PER_LINE` empirically against real font, or
- Switch LOADING to a dedicated render path that uses pixel-position
  alignment via container properties rather than space-padding inside
  text content
- Tune the vertical layout so WANDER sits visually centered top-half
  with the message close beneath the rule

Low-priority cosmetic fix. Doesn't block submission readiness.

### 3.4 [LOW] Minimap street/map overlay
**Steven's question (2026-04-26):** "Can the minimap show an overlay of
the street or map view?"

**Currently:** The minimap is a custom-encoded PNG generated from route
geometry only — just the path line, your position arrow, destination
marker. No street network underneath.

**Options for adding streets:**
1. **Static map tiles from OSM/Mapbox/Stadia/etc.** Fetch a small PNG
   centered on the route bounding box and render under the path.
   Mapbox Static and Stadia Maps both have free tiers.
2. **Render OSM road geometry from Overpass.** Pull `way[highway]`
   data inside the route bbox and rasterize the lines ourselves.
3. **Black-and-white OSM static styles** specifically tuned for
   monochrome displays.

**Recommendation:** v1.x feature, not v1.0. The G2 minimap area is
small (~240×240) and gray4 (16 levels), so high-detail tiles will look
muddy. A simplified style, monochrome custom render, or BBox-clipped
Mapbox static is the realistic path. Estimate: 1–2 sessions to do
well.
